CHANGELOG - Jinja2 Template Migration
======================================

WHAT CHANGED
------------

1. HTML files moved from pages/ to templates/
   Before: pages/home.html, pages/profile.html, etc.
   After:  templates/home.html, templates/profile.html, etc.
   Why: Jinja2 looks for templates in the templates/ directory by default.

2. Added a shared base template (templates/base.html)
   Before: every HTML file was a full standalone document, repeating
           <!DOCTYPE html>, <html>, <head>, <nav>, etc. each time.
   After:  base.html defines the shared layout once, and every other
           page just extends it and fills in its unique content.
   Why: avoids repetition. If you want to change the navbar, you only
        change it in one place instead of every single file.

3. HTML files now use Jinja2 syntax
   Before: plain static HTML with no dynamic content.
   After:  templates use {% extends %}, {% block %}, {{ variable }}, 
           {% if %}, {% for %} etc.
   Why: allows the server to inject data directly into the page before
        sending it to the browser (e.g. displaying the user's name on
        their profile page without needing a second fetch() call).

4. FastAPI routes updated to use TemplateResponse
   Before: return Path("pages/home.html").read_text()
   After:  return templates.TemplateResponse(request, "home.html")
   Why: TemplateResponse processes the Jinja2 template and fills in
        any variables before sending the HTML to the browser. The old
        approach just sent the raw file with no processing.

5. Profile route now fetches the user server-side
   Before: the profile page was static HTML, and JavaScript had to
           call fetch("/get_user?id=...") after the page loaded to
           get and display the user's data.
   After:  the /profile route receives the id, looks up the user, and
           passes it directly to the template. The page arrives in the
           browser already filled in.
   Why: simpler, faster, and less JavaScript to maintain.


WHY JINJA2?
-----------
Without Jinja2, any data that needed to appear on a page had to be
fetched by JavaScript after the page loaded (two round trips to the
server). With Jinja2, the server builds the complete page with all
the data already in it and sends it in one go. This makes the code
simpler and the pages load with their content immediately.

It also lets us share a common layout across all pages through
base.html, keeping the project organized as it grows.
